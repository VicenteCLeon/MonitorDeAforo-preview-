1. QUÉ ES ESTE SOFTWARE
────────────────────────────────────────────────────────────────────────────────
Sistema de monitoreo de aforo universitario en tiempo real para la PUCV.
Detecta personas en cámaras mediante YOLOv8 y muestra el estado en un dashboard web.

Stack:
  · Backend  → Python + FastAPI + YOLOv8 (ultralytics) + OpenCV + Supabase
  · Frontend → React 19 + TypeScript + Vite + Tailwind CSS v4
  · Auth     → Google OAuth (solo @mail.pucv.cl) + JWT propio del backend
  · BD       → Supabase (Postgres) — tabla "samples" + función "recent_occupancy"


────────────────────────────────────────────────────────────────────────────────
2. ARQUITECTURA ACTUAL (post-sesión 24-05-2026)
────────────────────────────────────────────────────────────────────────────────

FLUJO DE AUTENTICACIÓN:
  1. Frontend carga Google Sign-In SDK
  2. Usuario hace clic → Google devuelve id_token (JWT de Google)
  3. Frontend POST /api/auth/google { id_token }
  4. Backend valida el token contra tokeninfo de Google
     · Verifica aud == GOOGLE_CLIENT_ID
     · Verifica email_verified == true
     · Verifica que el email termine en @mail.pucv.cl
  5. Backend genera y devuelve un JWT propio (firmado con JWT_SECRET, expira 8h)
  6. Frontend guarda ese JWT en localStorage (clave: "monitor-aforo-token")
  7. Todas las llamadas al backend incluyen: Authorization: Bearer <jwt>
  8. El video stream usa ?token=<jwt> en la URL (los <img> no admiten headers)

ENDPOINTS DEL BACKEND:
  POST /api/auth/google       → público, rate-limited (10/min por IP)
  GET  /api/cameras           → protegido por JWT (header)
  GET  /api/counter/{cam_id}  → protegido por JWT (header)
  GET  /video_feed/{cam_id}   → protegido por JWT (query param ?token=)
  GET  /health                → público (solo info sin datos sensibles)

POLLING DEL FRONTEND:
  · Cada 2 segundos → fetchCameras(token) → actualiza conteos y sparklines
  · Cada 60 segundos → fetchHourlyOccupancy() → actualiza gráfico histórico (Supabase)
  · Cada 60 segundos → verifica expiración del JWT → logout automático si expiró

SPARKLINES (datos reales, no aleatorios):
  · App.tsx mantiene un historyRef (Map<camId, number[]>) con los últimos 24 conteos
  · Se acumula dentro del setFaculties callback en cada poll
  · Faculty.spark contiene el historial real que se pasa a FacultyCard → Sparkline
  · totalSpark (state) hace lo mismo para el KPI de aforo total
  · Los sparklines no aparecen hasta tener ≥ 2 lecturas reales

4. DECISIONES DE ARQUITECTURA TOMADAS EN ESTA SESIÓN
────────────────────────────────────────────────────────────────────────────────

· JWT en query param para video feed: los elementos <img src> no admiten headers
  personalizados, así que el token va en ?token=<jwt>. Es aceptable porque el
  token es de corta duración (8h) y la comunicación debe ir por HTTPS en producción.

· JWT_SECRET generado con secrets.token_hex(32): 64 chars hex = 256 bits de entropía.
  Si se cambia, todos los usuarios activos son deslogueados automáticamente.

· historyRef con useRef (no useState) para acumular el historial de sparklines:
  evita re-renders extra; el historial se incluye en el objeto Faculty que sí es state.

· genSpark eliminado completamente: los sparklines están vacíos al arrancar y se
  llenan con datos reales conforme avanza el polling. Honesto > bonito-pero-falso.

· ACTIVITY y FACULTIES_SEED eliminados de data.tsx: eran datos de ejemplo que
  podrían confundir; el log de actividad ahora se genera 100% desde eventos reales.

· CORS restringido a ALLOWED_ORIGINS (desde .env): en desarrollo = localhost:5173,
  en producción = el dominio real del frontend.

· Rate limiting solo en /api/auth/google (10/min por IP): es el único endpoint
  que puede ser abusado sin autenticación previa.

· /health es público pero no expone datos sensibles (solo cuenta de cámaras y flag
  de Supabase): útil para load balancers y monitoreo de infraestructura.

· Python 3.12 requerido para el backend: Python 3.14 no tiene wheels precompilados
  para torch/ultralytics/opencv → se cuelga al instalar. start.bat usa `py -3.12`.

· Alertas por email con lógica de transición de estado:
  - Los estados son "ok" / "warn" / "danger" (igual que el frontend: WARN_T=60%, DANGER_T=85%)
  - CameraWorker.prev_status rastrea el estado anterior por cámara
  - Solo se alerta cuando el estado ESCALA (ok→warn, ok→danger, warn→danger)
  - Las mejoras (danger→warn, warn→ok) no generan email
  - La ventana horaria (ALERT_HOUR_START/END) filtra alertas fuera del mediodía
  - /api/cameras expone "status" (calculado en tiempo real) en vez de "alert_active"

7. CÓMO ARRANCAR EL PROYECTO
────────────────────────────────────────────────────────────────────────────────

  REQUISITO: Python 3.12 instalado (no 3.14 — no tiene wheels para torch/ultralytics)
    · Instalar con: winget install Python.Python.3.12
    · start.bat usa `py -3.12 -m venv venv` explícitamente

  1. Doble clic en start.bat
     (crea venv con Python 3.12 e instala dependencias si faltan)

  2. Frontend: http://localhost:5173
     Backend:  http://localhost:8000
     API docs: http://localhost:8000/docs

  Variables de entorno necesarias:
    backend/.env  → JWT_SECRET, GOOGLE_CLIENT_ID, ALLOWED_ORIGINS
                    + SMTP_* para alertas (opcional, ver backend/.env.example)
    frontend/.env → VITE_GOOGLE_CLIENT_ID, VITE_SUPABASE_URL,
                    VITE_SUPABASE_ANON_KEY, VITE_API_BASE

  Si cambias JWT_SECRET → todos los usuarios activos pierden la sesión (esperado).

  Para probar alertas fuera del horario 13:30–14:30, cambiar temporalmente en main.py:
    ALERT_HOUR_START = (0, 0)
    ALERT_HOUR_END   = (23, 59)
